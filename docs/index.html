Key Features

Every data structure and algorithm is implemented from scratch — no third-party libraries, no frameworks. Each component reflects deliberate design choices grounded in academic literature and production engineering.

🔁
Three-Tier Deduplication
Bloom → Cache → Persistent
  • Probabilistic Bloom filter gates 99%+ of duplicate events at O(1)
  • LRU cache with TTL provides fast exact-match for recent events
  • Persistent store (pluggable) handles long-tail cross-session dedup
  • DeduplicationFilter composes all three tiers with short-circuit logic
sealed interface DeduplicationResult permits
    Duplicate, New, ProbablyNew // tier result

if (bloom.mightContain(key)) {       // tier 1: O(1)
    if (cache.contains(key)) ...       // tier 2: O(1)
    return persistent.check(key);      // tier 3: disk
}
🧮
Bloom Filter
MurmurHash3 · Lock-Free CAS
  • MurmurHash3 implemented from scratch — optimised 128-bit mixing pipeline
  • Lock-free bit manipulation via AtomicLongArray + CAS — no mutex contention
  • Double hashing (Kirsch–Mitzenmacher 2006): simulates k hashes with 2
  • Configurable false-positive rate with automatic optimal bit-size calculation
// double hashing: k probes from 2 hash functions
for (int i = 0; i < numHashFunctions; i++) {
    long h = hash1 + (long) i * hash2;
    int bit = (int) Math.floorMod(h, bitCount);
    bitArray.getAndUpdate(bit / 64,
        v -> v | (1L << (bit % 64)));
}
💾
LRU Cache + TTL
O(1) Access · ReentrantLock
  • Doubly-linked list + ConcurrentHashMap — classic O(1) get/put/evict
  • Sentinel head/tail nodes eliminate null-pointer branches in relink logic
  • Per-entry TTL with lazy expiration on access — no background thread needed
  • ReentrantLock guards structural mutations; read path is lock-free where safe
// O(1) eviction via doubly-linked list + sentinel nodes
private void moveToFront(Node node) {
    node.prev.next = node.next;   // unlink
    node.next.prev = node.prev;
    node.next = head.next;        // relink at front
    node.prev = head;
    head.next.prev = node;
    head.next = node;
}

Rule Engine AST
Sealed Types · Recursive Descent
  • Sealed interface RuleNode — exhaustive pattern matching, no reflection
  • Recursive descent parser: expression grammar, operator precedence, AST construction
  • Short-circuit evaluation ordered by DataSource weight (cheap checks first)
  • EvaluationPlan pre-sorts nodes by cost; avoids runtime overhead on hot path
sealed interface RuleNode permits
    AndNode, OrNode, NotNode, ConditionNode

return switch (node) {
    case AndNode(var l, var r) ->
        eval(l) && eval(r); // short-circuit
    case OrNode(var l, var r)  ->
        eval(l) || eval(r);
    case ConditionNode c -> check(c);
};

Campaign Management
State Machine · A/B Testing
  • 5-state state machine: DRAFT → SCHEDULED → ACTIVE → PAUSED → COMPLETED
  • A/B testing via deterministic hash bucketing — reproducible, no random state
  • Budget enforcement with CAS atomic decrement — no double-spend races
  • UserLimitTracker enforces per-user campaign frequency caps
// deterministic A/B bucket via hash (no Random)
int bucket = Math.abs(
    userId.hashCode() * 31 + campaignId.hashCode()
) % 100;
return bucket < config.trafficPercent()
    ? Variant.TREATMENT : Variant.CONTROL;
🚀
Action Dispatch
Token Bucket · Virtual Threads
  • Token bucket rate limiter: per-service quotas with configurable refill rate
  • Exponential backoff with full jitter — avoids thundering herd on retry storms
  • Java 21 virtual threads for dispatch — millions of concurrent inflight actions
  • ActionPipeline chains: rate-limit → schedule delay → execute → retry
// exponential backoff with full jitter
long base = 100L * (1L << attempt);
long cap  = Math.min(base, MAX_DELAY_MS);
long jitter = (long)(Math.random() * cap);
Thread.sleep(jitter); // virtual thread — no blocking
⏱️
Hierarchical Timing Wheel
Varghese & Lauck 1987 · O(1)
  • Inspired by Varghese & Lauck 1987 — O(1) insert and per-tick expiration
  • Wheel-of-wheels overflow hierarchy: minutes → hours without precision loss
  • Bucket-chained linked lists; tick advances pointer by one slot each tick
  • Used by DelayScheduler to schedule deferred action execution
// O(1) insert: slot = (currentTick + delay) % wheelSize
int slot = (int)((currentTick + delayTicks) % WHEEL_SIZE);
if (delayTicks >= WHEEL_SIZE) {
    overflow.add(task, delayTicks - WHEEL_SIZE);
} else {
    buckets[slot].add(task);  // O(1)
}
📄
JSON Codec
RFC 8259 · ~500 LOC · Zero Deps
  • Hand-rolled recursive descent parser conforming to RFC 8259 strictly
  • MurmurHash-keyed object representation — O(1) field lookup without sorted maps
  • ~500 lines of pure Java — no code generation, no annotation processing
  • Handles nested objects, arrays, escaped strings, numbers, nulls, booleans
// recursive descent: value → object | array | string | …
private JsonValue parseValue() {
    return switch (peek()) {
        case '{' -> parseObject();
        case '[' -> parseArray();
        case '"' -> parseString();
        default -> parsePrimitive();
    };
}

Design References

Each core data structure is grounded in peer-reviewed literature. Implementation choices are deliberate, not cargo-culted.

[1]
Hashed and Hierarchical Timing Wheels
Varghese & Lauck · ACM SIGCOMM 1987
Foundation for the TimingWheel implementation. Describes O(1) insert and expiration for timer-based systems using wheel-of-wheels overflow hierarchies. The paper defines the exact bucket-chaining algorithm used in DelayScheduler.
[2]
Space/Time Trade-offs in Hash Coding with Allowable Errors
Burton H. Bloom · CACM 1970
Original Bloom filter paper. Defines the probabilistic membership test structure used in BloomPrefilter. Provides the formula for optimal bit count and hash function count given a target false-positive rate.
[3]
Less Hashing, Same Performance: Building a Better Bloom Filter
Kirsch & Mitzenmacher · ESA 2006
Proves that double hashing g(x) = h₁(x) + i·h₂(x) achieves the same asymptotic false-positive rate as k independent hash functions. This eliminates k separate MurmurHash invocations in BloomPrefilter.
[4]
The JavaScript Object Notation (JSON) Data Interchange Format
RFC 8259 · IETF 2017
Normative specification for JSON syntax, encoding rules, and string escape sequences. JsonCodec implements a strict conforming parser — all edge cases (Unicode escapes, number precision, nested structures) handled per the RFC grammar.
